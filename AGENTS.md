# Repository Guidelines

## Project Structure & Module Organization
- `src/`: TypeScript source (e.g., `index.ts`, `gitlab-api.ts`, `schemas.ts`, `formatters.ts`, `transport.ts`). Tests live next to source as `src/*.test.ts`.
- `dist/`: Compiled JavaScript output (generated by `npm run build`).
- `assets/`: Images and repo assets.
- `docs/`: Public documentation, including `docs/api/` for the tool reference.
- Root config: `tsconfig.json`, `package.json`, `vitest.config.ts`, `README.md`.

## Build, Test, and Development Commands
- `npm run dev`: Start in dev mode with `ts-node` (ESM) using `src/index.ts`.
- `npm run build`: Compile TypeScript to `dist/` and mark entry as executable.
- `npm start`: Run the compiled server from `dist/index.js`.
- `npm test`: Run the vitest suite (use `npm run test:watch` for watch mode).
- Type-check only: `npx tsc --noEmit`.

## Coding Style & Naming Conventions
- Language: TypeScript (ES2022, `module`/`moduleResolution: NodeNext`, strict mode).
- Indentation: 2 spaces; keep lines focused and readable.
- Filenames: kebab-case (e.g., `gitlab-api.ts`, `formatters.ts`).
- Identifiers: `camelCase` for variables/functions, `PascalCase` for types/interfaces, `UPPER_SNAKE_CASE` for env vars.
- Imports: ESM paths with explicit `.js` when importing compiled files.

## Testing Guidelines
- Test framework: [vitest](https://vitest.dev/) with globals enabled and Node environment. Config in `vitest.config.ts`.
- Colocate tests next to the source as `src/*.test.ts` (see `src/schemas.test.ts`, `src/formatters.test.ts`, `src/gitlab-api.test.ts`).
- Keep tests deterministic. For network code, stub `node-fetch` via `vi.mock('node-fetch', () => ({ default: vi.fn() }))` rather than hitting a live GitLab.
- Cover error paths (invalid schemas, read-only mode filtering, transport setup) alongside the happy path.
- `npx tsc --noEmit` is useful locally for a quick type check; CI must run `npm test` to catch behavior regressions.

## Commit & Pull Request Guidelines
- Commit style: Conventional Commits (e.g., `feat: …`, `fix: …`, `chore: …`, `refactor: …`, `docs: …`).
- PRs should include: concise summary, motivation, linked issues, usage notes, and any config changes (env vars, ports).
- Update `README.md` when behavior or public tools change; include examples.
- Keep changes focused; prefer small, reviewable PRs.

## Security & Configuration Tips
- Required env: `GITLAB_PERSONAL_ACCESS_TOKEN`; optional: `GITLAB_API_URL`, `PORT`, `USE_SSE`, `GITLAB_READ_ONLY_MODE`. See `.env.example` for the template.
- Do not commit secrets. Use your shell or editor secrets store, not the repo.
- Validate changes in both transports: stdio (default) and SSE (`USE_SSE=true`).
- For safe demos, enable `GITLAB_READ_ONLY_MODE=true` to expose only read tools.

## Public Tools Summary
- `get_file_contents`: Reads a single file (file only; not a directory).
- `list_directory` (read-only): Lists repository contents via `/repository/tree` with `ref`, optional `path`, `recursive`, `page`, `per_page`.
- `push_files`: Accepts an array of file operations and creates a single commit.
  - Operation shape: `{ path: string, action: "create" | "update" | "delete", content?: string }`.
  - `content` is required for `create`/`update`, omitted for `delete`.
- JSON Schemas exposed to clients include `required` fields to improve tool UX; server-side validation remains authoritative via zod.
